## Day40——Git

### Git常用命令

1. **git clone——下载仓库内容**

   ```shell
   #下载远程仓库的内容，并且在本地创建一个和远程仓库名同名的文件夹，下载的时候, 路径在哪, 就直接下载到那个地方
   git clone https://gitee.com/boldness2012/xxxx.git
   
   #下载到指定的文件夹中(完全不建议)
   git clone https://gitee.com/boldness2012/xxxx.git  dirName
   ```

2. **git status——查看工作区和缓冲区和本地仓库中的变化**

3. **git add——把工作区中的变化提交到缓冲区**

   ```shell
   #所有文件: 把所有文件的变化都从工作空间提交到缓冲区
   git add .
   
   #文件的名字:  把指定的文件名文件在工作空间发生的变化提交到缓冲区
   git add fileName
   
   #文件的类型: *通配符
   git add *.c  #把所有以.c结尾的文件的变化, 从工作空间提交到缓冲区
   ```

4. **git commit——把缓冲区变化内容提交到本地仓库**

   ```shell
   #提交
   git commit -m "备注/注释"
   
   ## msg:msg信息一般要有统一的格式和意义(参照公司怎么写), 反正不建议乱写和简写(一定要注意) 
   ## 例如：Auther:ciggar Date:20220528 msg:xxx
   ```

5. **git log——查看仓库中的所有的版本信息**

   ```shell
   #列出版本的详细信息
   git log
   
   #列出压缩格式的所有分支的提交树 (--oneline: 每个版本压缩成一行)(--all:所有分支)(--graph:以分支树的形式展示)
   git log --oneline --all --graph
   
   #如果出现历史记录过长: Q回车退出
   ```

6. **分支操作——分支操作之前：需要确认工作空间和缓冲区干净状态, 以避免产生不必要的冲突**

   1. **查看分支**

      ```shell
      #查看所有分支:
      git branch -a
      
      #查看当前分支
      git branch   # 注意带 *
      ```

   2. **创建分支**

      ```shell
      #创建一个本地分支:
      git branch 分支名
      ```

   3. **切换分支**

      ```shell
      # 从当前分支切换到另一个分支
      git checkout 分支名
      
      #创建并切换分支: (等价于先创建, 再切换到创建的这个分支上)
      git checkout -b 分支名
      ```

   4. **删除分支**

      ```shell
      #安全的删除: (确保这个分支的内容分支已经被合并到上游)
      git branch -d 分支名
      
      #强制删除分支: 
      git branch -D 分支名
      ```

   5. **合并分支**

      ```shell
      #合并一个其它分支的内容到当前分支
      git merge 其它分支名
      
      #注意: 合并分支应该要向下游版本合并, 而非向上游合并
      #注意: Y型分支
      #假设存在A和B两个Y型分支:
      # 如果让在A上合并了B分支, 则A产生并演进到新的分支下游版本
      #如果想让B的版本进度也演进到刚才A产生的下游版本, 只需要把分支切回B, 并且对A进行merge, 则可以让AB都指向同一个版本
      
      #需要注意的是: 如果在合并过程中产生了无法自动merge的内容冲突, 需要手动处理冲突
      #当B分支被合并到A分支上, 如果不想继续使用B分支, B分支可被删除, B的删除不影响git记录保存B曾经产生的修改
      ```

7. **==冲突问题及其注意事项==**

   > 修改同一个文件,**导致内容冲突**: 需要手动合并(git对这种情况就无法自动合并产生合并版本了), 如果进行手动的冲突合并, 一定要记得需要重新add -> commit -> push（如果产生冲突, git能自动合并的的冲突不用做任何处理, 合并之后别忘了push,需要处理的是git不能自动合并的冲突, 重新先pull再add commit再push ）
   >
   >   冲突的约定俗成规则（每个人创建属于自己的分支）
   > (1)先push的人不处理冲突，后push的人要处理冲突(谁遇到冲突, 谁解决);
   >
   > (2)和组员一起开发的时候，尽量不要开发同一个文件，很容易产生冲突;
   >
   > (3)准备提交代码之前最好先pull一下;
   >
   > (4)早上上班之后，第一件事情，拉取最新的代码(pull)，晚上下班之前，最后的一件事情，把最新的本地代码推送上去（push），如果今天下班了, 代码没写完, 就不要push, 不要push

8. **提交与更新**

   1. **git push——把本地仓库中的版本变化推送到远程仓库**

      ```shell
      #把本地当前所在的分支的版本变化推送到跟踪/关联的远程分支上(但是易受git的设置的影响改变推送规则)
      git push
      
      #把本地当前所在的分支的版本变化推送到显示指明的远程分支上(不意味着可以乱推送)
      git push origin 分支名
      
      #如果本地创建了一个分支, 而远程不具有这个分支, 也可以通过命令的方式把它推送到远程上, 让远程多一个新分支 
      #提交分支: 在创建分支后第一次push的时候需要建立本地分支和远程分支之间的联系
      git push --set-upstream origin 分支名 
      
      #因为本地分支的删除, 并不会影响到远程的关联分支删除, 所以要删除远程分支, 除了远程仓库直接删除, 也可以在本地通过命令删除，可以通过命令删除远程分支
      git push origin --delete 分支名
      ```

   2. **git pull——拉取远程仓库中的所有的版本变化到本地**

      ```shell
      #获取远程仓库的更改(这个命令的本质: 是将远程分支的最新状态下载到本地仓库, 作为本地对远程版本的信息追踪记录)
      git fetch  origin 分支名
      (git fetch origin 分支名:分支名  ->  从远程仓库中获取指定分支的最新内容到本地仓库对应的分支,不存在分支则创建  )
      
      #把fetch下来的远程分支版本, 合并到当前版本中(要以分支命名和版本做匹配/关联分支, 不要乱merge) (注意斜杠, 不是空格)
      git merge origin/分支名  
      
      #把远端的关联的分支fetch下来,并且merge到当前所在分支上 
      git pull origin 分支名
      
      #如果之前已经建立关联分支, 等价于(git pull origin 分支名)
      git pull
      
      #注意:当本地仓库中的版本落后于远程仓库的时候，就要pull，落后就要pull (工作中一定要注意)
      ```

9. **回退操作**

   ```shell
   #回退某个被追踪文件在工作空间的修改 (未进行add和commit) (回退的内容，是找不回来的，要慎用)
   git checkout 文件名  
   
   #(注意:git status提示): git restore 文件名
   
   #把缓冲区中的变化，回退到工作空间:
   git reset 文件名
   
   git restore --staged 文件名   
   
   #回退本地版本
   git reset --hard  版本号
   
   #永远不要手动回退远程仓库的代码
   ```

    ```shell
    #通过上述流程:

    ```

### 笔记

1. git中Y型分支是什么？哪些情况下会有问题?如何解决Y型分支的冲突

   ```shell
   总结
   Y 型分支：主干切出分支后，主干和分支都有新提交，形成 Y 字形分叉；
   问题根源：两个分叉分支修改了同一代码，导致 Git 无法自动合并；
   解决方法：
   快速解决：git merge + 手动改冲突；
   规范协作：git rebase（消除 Y 型分叉）+ 手动改冲突；
   安全底线：只 rebase 私有功能分支，绝不 rebase 公共主干。
   ```

   从主干分支（`main`/`master`）检出一个功能分支后，**主干分支又新增了提交**，最终功能分支合并回主干时，两个分支的提交历史形成 **「Y」字形分叉 **

   ```shell
   # 初始状态：主干有A→B→C，从B切出功能分支
   main:    A --- B --- C
                \
   feature:      D --- E  # 功能分支开发
   
   # 主干新增提交F，形成Y型分叉
   main:    A --- B --- C --- F
                \
   feature:      D --- E
   
   # 合并后：Y型结构定型
   main:    A --- B --- C --- F --- M  (M=合并提交)
                          \       /
                           D --- E
   ```

   只有当**Y 型的两个分叉分支（主干 + 功能分支）修改了同一代码**，Git 无法自动合并，才会触发**合并冲突**（核心问题）

   **具体冲突场景**

   1. **核心代码冲突**：两个分支修改了**同一文件的同一行代码**
   2. **关联代码冲突**：两个分支修改了**同一文件的相邻行代码**（Git 判定为关联修改）
   3. **文件操作冲突**：主干删除文件，功能分支修改该文件；或一个分支重命名文件，另一个分支修改原文件
   4. **隐性逻辑冲突**：Git 自动合并无报错，但代码逻辑矛盾（如变量名、函数调用不匹配），导致运行报错

   Git 提供两种标准方案，**推荐用「变基 (rebase)」**（消除 Y 型分叉，提交历史更整洁）；也可以用「直接合并 (merge)」

   **方案 1：直接合并（git merge）- 快速解决，保留 Y 型结构**

   适合临时、简单的合并，步骤：

   1.切换到主干，拉取最新代码

   ```
   git checkout main
   git pull origin main
   ```

   2.合并功能分支，触发冲突

   ```
   git merge feature
   # 终端提示：Automatic merge failed; fix conflicts and then commit the result.
   ```

   3.手动解决冲突

   ```
   <<<<<<< HEAD        # 下方是主干(main)的代码
   主干修改的业务代码
   =======             # 分隔两个分支的冲突代码
   功能分支修改的业务代码
   >>>>>>> feature     # 下方是功能分支的代码
   ```

   ✅ 操作：删除标记（`<<<<<`/`=====`/`>>>>>`），保留**正确的业务代码**，保存文件。

   4.标记冲突解决，完成合并

   ```
   git add 冲突文件名  # 多个文件用 git add .
   git commit          # 自动生成合并提交
   ```

   5.推送到远程

   ```
   git push origin main
   ```

   **方案 2：变基合并（git rebase）- 消除 Y 型分叉，团队协作首选**

   #### 核心原理

   把功能分支的所有提交**平移**到主干最新提交的后面，将 Y 型分叉变成**线性提交历史**，无多余合并节点，主干历史更清晰。

   ⚠️ 黄金规则：**只对自己的私有功能分支 rebase，永远不要 rebase 公共主干（main/develop）！**

   步骤：

   1.切换到功能分支，拉取主干最新代码

   ```
   git checkout feature
   git fetch origin main  # 仅拉取不合并，安全无风险
   ```

   2.执行变基，嫁接分支

   ```
   git rebase origin/main
   # 有冲突时，rebase会自动暂停，提示解决冲突
   ```

   3.解决冲突（和 merge 完全一致）

   编辑文件 → 删除冲突标记 → 保存

   4.标记冲突解决，继续变基

   ```
   git add 冲突文件名
   git rebase --continue  # 继续执行变基
   ```

   5.重复 3-4 步，直到所有冲突解决（变基完成）

   6.切换到主干，**快进合并**（无冲突、无新提交）

   ```
   git checkout main
   git merge feature  # 快进合并，直接生效
   git push origin main
   ```

   **紧急回退（操作失误必用）**

   - 合并中途放弃：`git merge --abort`
   - 变基中途放弃：`git rebase --abort`

   **关键注意事项**

   1. 解决冲突**不要盲目删代码**，务必和对应开发确认业务逻辑；
   2. 冲突解决后**必须测试代码**（编译 / 运行），避免隐性逻辑冲突；
   3. 团队协作优先用`rebase`整理功能分支，再合并主干，保持主干线性清晰；
   4. Y 型分支是 Git 正常模式，**冲突是并行开发的必然结果**，按步骤解决即可。

2. 多人合作写项目的时候，如何防止冲突？如果发生冲突，如何解决

   **一、多人协作：如何最大限度防止冲突？**

   核心思路：**减少「多人同时修改同一文件 / 同一行代码」**，靠「团队规范」+「个人操作习惯」双管齐下。

   **1. 团队级规范（从根源杜绝大部分冲突）**

   1. **严格的分支规则（最重要！）**

      绝对禁止所有人直接修改 `main`（主干）/`develop`（开发分支）！

      - 公共分支：`main`（上线代码）、`develop`（开发汇总）

      - 个人分支：每个人新建专属功能分支（如 feature/login-张三、fixbug/支付报错-李四），自己的分支自己用

        ✅ 效果：大家各写各的，不会直接互相干扰。

   2. **代码模块化，分工明确**

      不要多人挤在同一个文件里开发：

      - 前端：分页面、分组件开发，不随意改别人的组件

      - 后端：分接口、分服务开发，不随意改公共工具类 / 配置文件

        ✅ 效果：从业务上避免「同文件修改」。

   3. **修改公共代码必须沟通**

      公共文件（工具类、配置文件、公共样式）是冲突重灾区！

      改之前在群里说一声，改完及时同步给大家。

   4. **小步提交，不要憋大招**

      不要写几百行代码才提交一次，**功能写完一小块就提交**，提交越频繁，冲突越小。

   ------

   **2. 个人操作习惯（每天必做，零成本防冲突）**

   这是**最有效、最简单**的预防手段，所有人都要遵守：

   1. 每天开工第一件事：拉取最新代码

      写代码前，先同步公共分支的最新代码，让你的本地代码永远是最新的：

      ```
      git checkout develop
      git pull origin develop
      ```

   2. 频繁同步

      开发中途，每隔几小时就拉一次最新代码，不要一天只拉一次。

   3. 绝不碰公共分支

      只在自己的功能分支上写代码，永远不直接在 develop/main上修改。

   ------

   **3. 技术防护（进阶，团队仓库设置）**

   在 Git 仓库后台（Gitee/GitHub/GitLab）开启保护：

   - 禁止直接推送代码到 `main/develop`

   - 所有代码必须通过 PR/MR（合并请求） 提交，经过同事审核后才能合并

     ✅ 效果：强制规范流程，彻底杜绝乱改主干的情况。

   ------

   **二、发生冲突了？3 步傻瓜式解决（新手也会）**

   冲突的本质：**你和同事改了同一行代码，Git 不知道该留谁的**，会暂停合并，让你手动选择。

   **通用原则**

   1. 冲突不是 BUG，是正常现象，不用慌
   2. 解决冲突**必须和同事沟通**，不要盲目删代码
   3. 解决完一定要运行代码测试

   ------

   **场景 1：最常见 → 拉代码时冲突（git pull）**

   你在写代码，远程有同事更新了代码，你拉取时报错冲突：

   **步骤**

   1. 查看冲突文件

      终端会提示：Automatic merge failed，并列出冲突的文件名。

   2. 手动修改冲突文件（核心）

      打开文件，Git 会用固定标记标出冲突区域：

      ```
      <<<<<<< HEAD        # 这是你本地的代码
      你写的代码
      =======             # 分隔线
      同事写的代码
      >>>>>>> develop     # 这是远程公共分支的代码
      ```

      ✅ 操作：

      - 删除 `<<<<<` / `=====` / `>>>>>` 这些标记
      - 和同事沟通，保留**正确的业务代码**（可以都保留、只留你的、只留同事的）
      - 保存文件

   3. 完成冲突解决

      ```
      git add 冲突文件名   # 标记冲突已解决
      git commit -m "解决冲突：合并xxx文件"
      git push            # 推送到远程
      ```

   ------

   **场景 2：功能分支合并到公共分支时冲突（PR/merge）**

   你的分支开发完了，想合并到 `develop`，提示冲突：

   **推荐方案：变基（git rebase）→ 代码历史最干净**

   1. 切换到你的功能分支

      ```
      git checkout feature-xxx
      ```

   2. 把公共分支最新代码同步到你的分支

      ```
      git fetch origin develop
      git rebase origin develop
      ```

   3. 有冲突就按场景 1 的方法修改，然后继续

      ```
      git add 冲突文件
      git rebase --continue
      ```

   4. 冲突全部解决后，重新提交 PR → 无冲突直接合并

   **简单方案：直接合并（git merge）**

   1. 切换到公共分支，拉最新代码
   2. 合并你的功能分支
   3. 解决冲突 → 提交 → 推送

   ------

   **3. 操作失误？一键回退**

   如果解决冲突时改乱了，想放弃操作：

   - 合并冲突放弃：`git merge --abort`
   - 变基冲突放弃：`git rebase --abort`

   ------

   **4. 新手友好：不用记命令的可视化工具**

   如果不想敲代码，用工具一键解决冲突：

   - VS Code（自带冲突解决界面，点一点就能选保留谁的代码）
   - SourceTree / GitKraken（图形化 Git 工具，全程鼠标操作）

   ------

   **三、避坑红线（千万不要做）**

   1. **不要强行覆盖别人的代码**：冲突必须沟通，否则会导致功能失效
   2. **不要 rebase 公共分支**：只能 rebase**自己的功能分支**，否则会搞乱团队代码
   3. **解决冲突后不测试**：Git 只能解决语法冲突，**逻辑冲突**要自己测（比如函数名改了、变量删了）

   ------

   **总结**

   1. **防冲突核心**：分支规范 + 每天拉最新代码 + 分工明确，能解决 90% 的冲突；
   2. **解冲突核心**：找到 Git 标记 → 删除标记、保留正确代码 → 提交同步；
   3. **团队底线**：不直接改主干、改公共代码先沟通、冲突必确认。

   按照这个方法，多人协作基本不会被冲突困扰！
